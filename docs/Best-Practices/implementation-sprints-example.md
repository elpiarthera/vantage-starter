## ЁЯОп SPRINT STRATEGY

### Core Principles

1. **Schema First**: All database schema changes happen in Sprint 0 to avoid mid-sprint migrations
2. **Vertical Slices**: Each sprint after 0 delivers a complete, testable feature (backend + admin UI + public UI)
3. **MVP Focus**: Build only what's defined in the architecture; defer "nice to have" features
4. **Hours, Not Weeks**: Each sprint is estimated in hours, targeting rapid iteration
5. **Independent Deployability**: Each sprint's output can be deployed and tested independently
6. **Quality Built-In**: Each sprint includes testing, linting, and regression prevention (no separate QA sprint)

### Team & Resources

- **Solo Developer**: One full-time developer with AI assistance (Claude, Grok, Gemini)
- **Work Schedule**: 7 days per week, highly focused sessions
- **Efficiency Multiplier**: AI-assisted development enables faster execution than traditional estimates
- **Note**: Estimates are conservative; actual completion may be 20-40% faster with AI pair programming

### Sprint Management

- **Detailed Plans**: Each sprint will have its own dedicated implementation document
- **Continuous Retrospectives**: Sprint plans updated after each task completion
- **Real-Time Adaptation**: Adjust scope and estimates based on actual progress
- **No Separate QA Sprint**: Quality assurance is built into every sprint

## ЁЯФз SPRINT 0: Database Schema & Core Foundation

**Goal**: Establish the complete database schema and core utilities so no schema changes are needed in future sprints.

**Duration Estimate**: 8-12 hours

**Why First**: 
- Convex schema changes are instant to deploy, so we can iterate quickly
- All future sprints depend on having the correct schema in place
- Prevents costly refactoring later
- Enables parallel development of features once schema is stable

### Main Tasks

**Database Schema (Convex)**:
- Create all 14 tables defined in architecture:
  - `verticalConfig` (key-value store for settings)
  - `resources` (universal resource table with customFields)
  - `categories` (dynamic categories)
  - `resourceClaims` (entity claiming workflow)
  - `entities` (universal entity/business table)
  - `promotions` (monetization system)
  - `entityMembers` (team management)
  - `aiPrompts` (admin-configurable prompts)
  - `scrapingJobs` (mass scraping)
  - `scrapedResourcesQueue` (review queue)
  - `resourceFieldDefinitions` (dynamic resource fields)
  - `entityFieldDefinitions` (dynamic entity fields)
  - `translations` (i18n system)
  - `auditLogs` (configuration tracking)

**Core Utilities**:
- Configuration loader (reads `verticalConfig` with caching)
- Field definition validator (validates customFields against definitions)
- Prompt resolver (handles inheritance: vertical тЖТ base)
- Translation helper (key lookup with locale fallback)

**Testing Infrastructure**:
- Seed script for base configuration
- Seed script for sample base prompts
- Test data generator for development

### Testing & Quality Assurance

**Type Safety**:
- Run `tsc --noEmit` to validate all TypeScript types
- Ensure schema types are correctly generated by Convex
- No type errors in core utilities

**Linting**:
- Run Biome (`biome check`) on all new files
- Fix formatting and linting issues
- Ensure consistent code style

**Unit Tests**:
- Test configuration loader (caching, fallbacks)
- Test field validator (valid/invalid cases)
- Test prompt resolver (inheritance logic)
- Test translation helper (locale fallback)

**Integration Tests**:
- Verify all tables exist in Convex
- Test seed scripts populate data correctly
- Verify indexes are created

**Regression Prevention**:
- Snapshot test of schema structure
- Ensure no existing functionality breaks

**Success Criteria**:
- тЬЕ All tables exist in Convex with correct indexes
- тЬЕ Schema validator runs without errors
- тЬЕ Seed scripts populate initial data
- тЬЕ Core utilities have basic tests
- тЬЕ No schema changes planned for future sprints
- тЬЕ TypeScript compiles without errors
- тЬЕ Biome passes all checks
- тЬЕ All tests pass

**Blockers**: None (this is the foundation)

- **Mitigation**: Architecture has been validated through 6 review rounds

## ЁЯУЭ NOTES & CONSIDERATIONS

### Technical Decisions

1. **Why Schema First**: Convex allows instant schema changes during development, but production migrations are risky. Doing all schema work upfront prevents mid-sprint refactoring.

2. **Why Vertical Slices**: Each sprint delivers a complete, testable feature rather than "backend sprint + frontend sprint." This enables continuous deployment and testing.

3. **Why This Order**: Dependencies dictate order. Configuration and fields are foundational; entities require fields; monetization requires entities; scraping requires fields and prompts.

### Scope Management

- **MVP mindset**: Build only what's in the architecture document
- **No scope creep**: New ideas go into "Future" list, not current sprints
- **Feature flags**: Use for incomplete features (don't block other work)

### Quality Standards

- **Each sprint must**:
  - Be deployable independently
  - Have basic testing (manual or automated)
  - Be documented (comments, README)
  - Work on both desktop and mobile

### Risk Mitigation

- **Schema errors**: Sprint 0 has extra buffer time for iterations
- **Integration issues**: Stripe/Firecrawl tested in isolation first
- **Performance**: Monitor Convex query performance, optimize as needed
